# 可持续二次开发与上游同步设计

## 设计目标

用稳定、可审查且无需改写共享历史的 Git 流程，将上游更新和自定义功能开发分离。生产部署只使用经过验证的 `custom` 分支。

核心原则：

1. **现在就能二开**：以当前可用 `custom` 为起点，不等 release。
2. **默认同步 release**：生产升级优先稳定 tag。
3. **允许按需中途同步**：release 太远或急需上游改动时，可临时同步，但必须审查、验证、可回滚，且不能变成默认节奏。
4. **代码边界清晰**：自定义逻辑优先落在新文件/新模块；对上游文件只做窄入口改动，并用可检索标记标注。
5. **扩展有优先级**：先配置/开关，再新模块注册，再包装，最后才改上游核心。

## 分支职责

| 分支 | 生命周期 | 职责 | 允许的变更 |
| --- | --- | --- | --- |
| `main` | 长期 | 镜像最近一次已接纳的上游稳定 release | 仅允许从已核验的上游 tag 快进 |
| `custom` | 长期 | 保存已验证、可部署的二开版本 | 已验收的功能合并和已验证的同步合并 |
| `feature/<name>` | 短期 | 隔离单个功能或修复 | 与该需求直接相关的小型提交 |
| `sync/vX.Y.Z` | 短期 | 集成并验证一个上游稳定 release | `main` 合并、冲突解决及必要兼容修复 |
| `sync/upstream-main-YYYYMMDD` | 短期 | 按需中途同步上游 `main`（例外路径） | 经审查的 merge / cherry-pick 与兼容修复 |
| `backup/<name>` | 临时 | 高风险操作前保存可恢复指针 | 不开发；确认稳定后删除 |

当前不设置永久 `dev`。只有在多人并行、多个功能需要联合验收，或存在独立集成环境时再引入。

## 当前状态与开工策略

当前 `main` 和 `custom` 位于 `eb2b8632`，比最新稳定版 `v0.1.156` 多 15 个上游提交。该状态**可直接作为二次开发起点**。

### 可立即执行

1. 从当前 `custom` 创建 `chore/trellis-workflow`，提交 `.trellis/`、`.cursor/`、`.agents/`（独立 chore，不含业务代码）。
2. 从最新 `custom` 创建 `feature/<name>` 开始正式业务功能。
3. 高风险操作前可选创建 `backup/*` 指针。

### 与 release 对齐（可稍后）

`main` 正式镜像到某个稳定 release tag **与功能开发解耦**，在下列时机执行即可：

1. 出现包含 `eb2b8632` 的新稳定 release（`eb2b8632` 是该 tag 的祖先）。
2. 将 `main` 以 `--ff-only` 快进到该 tag，并明确推送 `main`（禁止 `git push --tags`）。
3. 若当时 `custom` 尚无额外提交且与 `main` 历史兼容，可 ff；若 `custom` 已有二开提交，则用 `sync/vX.Y.Z` 合并验证后再回 `custom`。

如果目标 release 不包含 `eb2b8632`，停止并将该情况视为上游改写/分叉，重新评估；不得强行把错误 tag 设为 `main` 基线。

## 功能开发流程

1. 从最新 `custom` 创建 `feature/<name>`。
2. 每个提交只承担一个可解释的目的；避免无关格式化、生成文件和业务改动混合。
3. 开发期间可从 `custom` 更新功能分支；**不得**从 `upstream/main` 直接拉入开发中的 feature 分支。
4. 落点前先按「扩展优先级」选型（见下文），避免一上来改上游巨型 service / 大表单。
5. 完成代码审查和适用测试后，将功能分支合并回 `custom`。
6. `custom` 合并后必须保持可部署；未完成或未验证功能不得提前进入。

## 上游同步策略（分层）

### A. 默认路径：稳定 release

1. 获取上游分支和 tags，并从 GitHub Release 页面确认目标版本。
2. 审查当前基线到目标 tag 的提交、文件差异、依赖更新和数据库迁移。
3. 验证目标 tag 是当前 `main` 的后代，然后仅用 fast-forward 更新 `main`。
4. 从当前 `custom` 创建 `sync/vX.Y.Z`，把更新后的 `main` 以显式 merge commit 合入。
5. 只在 `sync/*` 中解决冲突和编写必要兼容修复，不直接污染 `main`。
6. 运行后端、前端、构建及部署相关检查；涉及数据库迁移时先完成备份和恢复演练。
7. 验证通过后将 `sync/*` 快进合并回 `custom`，再推送；失败则删除同步分支，`custom` 保持不变。

### B. 例外路径：按需中途同步

适用条件（满足其一即可评估）：

- 下一个稳定 release 预计较远，但上游已有你急需的修复或功能。
- 线上问题已被上游修掉，等待 tag 成本过高。

执行约束：

1. 从 `custom` 创建 `sync/upstream-main-YYYYMMDD`（或仅 cherry-pick 目标提交到临时分支）。
2. **先审查**范围：提交列表、diff、依赖、`backend/migrations/`。
3. 不把 `upstream/main` 直接合进长期 `main`（`main` 仍只镜像稳定 release）。
4. 在 `sync/*` 完成冲突处理、兼容修复与完整验证。
5. 验证通过后再合回 `custom`；记录来源（commit/range）与原因。
6. 下次正式 release 同步时，把中间态变更纳入 release 审查，避免重复冲突与遗漏。

禁止事项：

- 无审查地长期跟踪 `upstream/main` 作为默认节奏。
- 在 feature 分支上直接 `merge upstream/main`。
- force push / 改写共享历史。


## 代码级二次开发约束

目标：把自定义与上游的冲突面压到最小，使 `sync/*` 审查可定位、可回滚，而不是每次升级重写补丁。

### 扩展优先级（必须按序评估）

| 优先级 | 方式 | 适用 | 对 sync 的影响 |
| --- | --- | --- | --- |
| 1 | 配置 / 特性开关 / 环境变量 | 行为可开关、无需新领域模型 | 最低：通常不改核心逻辑 |
| 2 | 新模块 + 薄注册点 | 新 API、新页面、新 service 能力 | 低：冲突多集中在路由/DI/导航入口 |
| 3 | 包装 / 装饰现有 service | 需横切已有流程且不宜 fork 核心 | 中：入口文件有小 diff |
| 4 | 修改上游核心逻辑 | 仅当 1–3 无法满足 | 高：必须最小化 diff 并加标记 |

禁止默认选择优先级 4。若必须改核心，PR/提交说明中写清：为何更低优先级不可行、改了哪些上游文件、如何验证。

### 代码落点规则

1. **优先新增文件**：backend 新能力放独立 package/file（如新 handler、service、repository）；frontend 新页面/组件/API 模块独立文件。
2. **上游文件只做窄改**：仅改必要挂点——路由注册、DI 装配、导航菜单、配置 schema、迁移注册表等；不在巨型函数中心堆业务分支。
3. **冲突高发区克制**：
   - 超大 Vue 页面 / 巨型 service / 共享 schema 生成物；
   - 上游频繁迭代的 OAuth/billing/gateway 核心路径；
   - 已有大段逻辑的中间位置插入（优先文件末尾注册或独立函数）。
4. **跨层改动成套**：若新增 API，同时规划 handler → service → repository → migration → frontend API/类型/页面；避免只改一层留下半成品。
5. **禁止「顺手重构」上游代码**：格式化、重命名、无关提取不得与功能提交混做；这类改动会显著放大 sync 冲突。

### 二开标记约定

在**被修改的上游原有文件**内，用统一可检索标记包裹或标注自定义点：

- Go：`// CUSTOM: <简述>` 或 `// CUSTOM:begin` … `// CUSTOM:end`
- Vue/TS/JS：`// CUSTOM: <简述>` 或 `{/* CUSTOM: <简述> */}`
- SQL migration（自定义文件头）：`-- CUSTOM: <简述>`

规则：

- 新增的纯自定义文件：文件头一条 `CUSTOM` 说明即可，不必逐行标记。
- 修改上游文件：每个逻辑插入点至少一条标记，便于 `rg "CUSTOM:"` 在 sync 前列出全部分叉点。
- 标记描述写「为什么存在」，不写废话；删除自定义时同步删除标记。

### 自定义 migration 策略

- 项目迁移为 **forward-only**；回滚依赖备份或补偿迁移，不依赖 down。
- 自定义迁移文件命名与上游隔离，避免与下一版上游 release 文件名/序号冲突。推荐前缀或后缀体现 fork 身份，例如：
  - `YYYYMMDDHHMMSS_custom_<slug>.sql`（若上游使用时间戳）
  - 或项目既有命名风格下增加明确 `custom_` 段
- **禁止**修改、重排、改写上游已有 migration 文件内容或 checksum。
- 每次 sync 审查清单必须包含：`backend/migrations/` 新增/变更列表；自定义迁移与上游新迁移的顺序兼容性。
- 生产升级前：备份 → 在演练环境跑通迁移 → 再升 `custom` 构建。

### 依赖边界

- 新增 Go module / npm 依赖：单独提交或在功能提交说明中写明用途；优先复用已有依赖。
- 不主动大版本升级上游已锁定依赖，除非：
  - 本次 `sync/*` 上游已升级；或
  - 安全漏洞且无更低风险替代。
- lockfile 变更必须与真实依赖变更同提交，禁止无意义 lockfile churn。

### 提交分类

| 类型 | 允许内容 | 禁止混入 |
| --- | --- | --- |
| `feat` / `fix`（feature 分支） | 单一业务意图的代码与测试 | workflow 目录、无关注释格式化、上游大范围整理 |
| `fix`（sync 兼容） | 仅为吸收上游而必需的冲突解决与适配 | 新业务功能 |
| `chore`（workflow） | `.trellis/`、`.cursor/`、`.agents/` 等 | 业务代码、依赖业务变更 |
| `chore`（deps） | 依赖与 lockfile | 功能逻辑 |

### 降低冲突的实践清单（开发时）

- [ ] 是否可用优先级 1–3 完成，而不改核心？
- [ ] 上游文件 diff 是否只含挂点级改动？
- [ ] 是否已加 `CUSTOM` 标记且 `rg "CUSTOM:"` 可列出？
- [ ] 自定义 migration 是否未触碰上游 migration？
- [ ] 提交是否单一意图、可独立 revert？

## 版本与发布边界

- 上游 tags 仅作为只读输入，不应使用 `git push --tags` 将其批量推送到 fork。
- fork 的 `.github/workflows/release.yml` 会被 `v*` tag 触发，因此在设计自定义镜像版本前不得随意推送 `v*` tag。
- 当前规划不创建自定义 release 或镜像。后续若使用 Docker 部署自定义代码，需要单独设计 fork 镜像命名、tag 规则和部署文件，避免继续拉取上游镜像。

## 回滚策略

- Git 集成失败：放弃 `feature/*` 或 `sync/*`，长期分支不受影响。
- 已合并但未部署：用新的 revert 提交撤销，不改写 `custom` 的共享历史。
- 已部署且无数据库变更：回退到上一个已验证的自定义构建。
- 已执行数据库迁移：项目迁移为 forward-only；必须依赖升级前备份恢复或经过验证的补偿迁移，不能只回退应用代码。

## 保护措施

- 为 fork 的 `main` 和 `custom` 启用分支保护，禁止 force push。
- `main` 禁止直接业务提交；`custom` 禁止未审核的直接开发。
- 推送时明确写分支名，避免 `--all` 和 `--tags`。
- 每次同步记录来源（release tag 或 commit range）、旧基线、测试结果和数据库迁移清单。
- 每次 sync 前运行 `rg "CUSTOM:"`（或等价检索）列出全部分叉点，对照冲突文件。
- Code review 对「修改上游核心文件」的 PR 提高审查门槛：必须说明为何未走扩展优先级 1–3。
