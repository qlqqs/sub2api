# 规划可持续二次开发与上游版本同步

## Goal

建立一套可长期维护的二次开发工作流，使自定义功能与上游代码在 **Git 流程与代码落点** 上边界清晰，并能持续、安全地吸收 `Wei-Shaw/sub2api` 更新；**默认同步稳定 release，允许按需中途同步**，且**不因等待 release 而阻塞二次开发**。

## Background

- Fork 远程 `origin` 为 `https://github.com/qlqqs/sub2api.git`。
- 原项目远程 `upstream` 为 `https://github.com/Wei-Shaw/sub2api.git`。
- 当前分支为 `custom`；本地 `main`、`origin/main`、`custom` 和 `upstream/main` 当前均指向提交 `eb2b8632`。
- `custom...upstream/main` 的左右差异计数为 `0 0`，当前尚无自定义提交需要迁移或解决冲突。
- 上游通过 `v*` tag 发布；当前最新稳定 release 为 `v0.1.156`。
- 当前 `upstream/main` 位于 `v0.1.156` 之后 15 个提交，因此跟踪 `main` 与跟踪稳定 release 是两种不同的升级策略。
- **决策更新（2026-07-16）**：当前代码已可用，**立即基于 `custom` 做二次开发**；不必等待下一个稳定 release 才开工。`main` 正式镜像到 release tag 可在后续合适时机完成，与功能开发解耦。
- **同步策略更新**：生产级升级**默认**以稳定 release tag 为输入；若 release 过远且急需上游修复/功能，允许经审查的临时同步（`upstream/main` 或 cherry-pick），不强制每次都等 release。
- `.agents/`、`.cursor/` 和 `.trellis/` 当前为未跟踪的本地工作流文件；纳入 fork 时必须用独立 `chore` 提交，不得与业务改动混在同一提交。

## Requirements

- 保留一个可直接对照上游的干净基线（`main`），避免将自定义功能直接堆积在同步基线上。
- 所有自定义功能在 `feature/*` 中开发，验收后合并进长期 `custom`；提交小而清晰。
- 分支模型：`main` 镜像已接纳的上游稳定 release；`custom` 承载可部署的二开版本；`sync/vX.Y.Z`（或临时 `sync/upstream-main-YYYYMMDD`）用于升级验证。
- `custom` 必须始终保持可部署；新功能使用短期 `feature/*`，当前不增加永久 `dev` 集成分支。
- **允许立即二次开发**：以当前可用的 `custom`（`eb2b8632`）为开发起点，不因「首次 release 基线尚未对齐」而阻塞业务功能。
- **默认同步目标是稳定 release**，不是无条件长期跟踪 `upstream/main`；但允许按需中途同步，且必须先审查、验证、可回滚。
- 禁止通过覆盖历史或无审查地拉取代码来完成生产升级。
- 数据库迁移必须作为升级检查重点；项目迁移为 forward-only，回滚依赖升级前备份或补偿迁移。
- `.trellis/`、`.cursor/` 和 `.agents/` 作为可复用的项目工作流配置纳入 fork，但必须使用独立 `chore` 提交，与业务改动隔离。
- 不回退长期分支，不强制推送远端共享历史。
- **代码级二开约束**：自定义功能优先新增文件/模块，再通过薄适配接入；尽量缩小对上游核心文件的修改面。
- **扩展优先级**：配置/特性开关 → 新模块+注册点 → 包装/装饰现有 service → 最后才修改上游核心逻辑。
- **二开标记可检索**：在改动的上游文件内使用统一标记（如 `// CUSTOM:` / `// FORK:` 或等价注释），便于 sync 时定位冲突与回归。
- **自定义迁移独立**：自定义数据库迁移与上游迁移在命名/编号策略上隔离，避免与上游 release 撞车；迁移仍为 forward-only。
- **依赖边界**：新增依赖须单独说明；尽量不升级上游已锁定的大版本依赖，除非同步本身要求。
- **提交意图单一**：业务功能、同步兼容修复、workflow chore 不得混在同一提交。

## Acceptance Criteria

- [ ] 明确 `main`、`custom` 和临时升级分支各自的职责。
- [ ] 明确默认同步目标是上游稳定 release；同时写清何时允许按需中途同步及其约束。
- [ ] 明确可以基于当前 `custom` 立即开始二次开发，首次 release 基线对齐与功能开发解耦。
- [ ] 给出首次建立分支模型与 workflow 提交的安全步骤，不丢失现有工作。
- [ ] 给出每次新 release 的获取、审查、集成、测试、推送和回滚流程。
- [ ] 给出按需中途同步（临时 `upstream/main` / cherry-pick）的流程与停止条件。
- [ ] 给出自定义功能的提交边界及降低冲突的原则。
- [ ] 明确 `.agents/`、`.cursor/`、`.trellis/` 的处理策略。
- [ ] 明确代码落点规则：优先新增文件/模块，窄改上游入口，冲突高发区克制堆业务。
- [ ] 明确扩展方式优先级（配置 → 新模块 → 包装 → 改核心）。
- [ ] 明确二开标记约定与检索方式，便于 sync 审查。
- [ ] 明确自定义 migration 命名/编号隔离策略及与上游迁移的审查要点。
- [ ] 明确依赖变更与提交分类边界（feature / sync 兼容 / chore workflow）。
- [ ] 规划执行阶段将上述约束沉淀为 `.trellis/spec/guides/` 下可复用规范（本阶段先写清要求，不强制已落盘）。

## Out of Scope

- 本规划阶段不修改业务代码（workflow 目录的 chore 提交除外，可在执行阶段落地）。
- 本规划阶段不创建自定义 release、Docker 镜像或生产部署。
- 在具体功能需求提出前，不设计功能级代码改动。
