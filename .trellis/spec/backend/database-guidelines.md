# 数据库规范

## 事实基线

- 生产数据库是 PostgreSQL。连接由 `backend/internal/repository/ent.go:InitEnt` 使用
  Ent PostgreSQL driver 建立；`backend/internal/repository/wire.go:ProvideSQLDB` 从同一个
  Ent driver 暴露底层 `*sql.DB`。
- 持久化是 **Ent + `database/sql` 混合模式**，不是纯 ORM，也没有 GORM。Ent 适合实体 CRUD、
  predicate、edge、upsert 和行锁；原生 SQL 用于复杂聚合、批量/条件更新、热路径和需要精确
  PostgreSQL 语义的原子操作。
- `backend/migrations/*.sql` 是运行中数据库 schema 的权威来源。
  `backend/ent/schema/**` 同时描述生成客户端所需的字段/edge，但应用启动不调用 Ent
  auto-migration。`InitEnt` 的注释和实现明确先运行 SQL migrations，再创建 `ent.Client`。
- Repository 实现 service 层定义的端口并返回 service model。真实例子：
  `backend/internal/repository/user_repo.go:NewUserRepository` 返回
  `service.UserRepository`；`backend/internal/repository/usage_billing_repo.go` 直接使用
  `*sql.DB` 实现 `service.UsageBillingRepository`。

## Schema 与生成代码

### 手写范围

- 实体定义放在 `backend/ent/schema/<entity>.go`。通过
  `entsql.Annotation{Table: "..."}` 显式绑定现有复数 snake_case 表名；参考
  `schema.User.Annotations`、`schema.APIKey.Annotations` 和 `schema.Account.Annotations`。
- 公共时间字段使用 `backend/ent/schema/mixins/time.go:TimeMixin`：`created_at` immutable，
  `updated_at` 带 update default，PostgreSQL 类型均为 `timestamptz`。
- 需要软删除的实体使用
  `backend/ent/schema/mixins/soft_delete.go:SoftDeleteMixin`。普通 Ent query 自动追加
  `deleted_at IS NULL`，delete mutation 转成更新时间；只有恢复、审计、物理清理等明确场景才
  使用 `mixins.SkipSoftDelete(ctx)`。真实用法见
  `repository.userRepository.GetByIDIncludeDeleted`。
- PostgreSQL 专有类型要在 schema 中显式 `SchemaType`，例如
  `schema.User` 的 `decimal(20,8)`、`schema.Account` 的 `jsonb`、
  `schema.UserSubscription` 的 `timestamptz`/`decimal(20,10)`。当前 service model 大量仍使用
  `float64`；新增字段应匹配邻近模型，不要误称 Go 层已经具备任意精度，也不要无迁移地改变
  数据库 precision/scale。

### SQL 与 Ent 必须一起演进

新增或修改持久化字段时通常要同时完成：

1. 更新 `backend/ent/schema/**`，使生成 builder/query/model 能访问字段。
2. 新增 `backend/migrations/*.sql`，使已部署数据库前向演进。
3. 运行 `make -C backend generate`，更新 Ent 生成代码及 Wire 生成代码。
4. 增加 repository 行为测试；重要列、索引、约束还应加入 migration/schema 集成断言。

不能只改 Ent schema：`InitEnt` 不执行 `client.Schema.Create`。也不能只改 SQL 后立即在代码中
引用新字段：生成的 Ent client 尚不知道该字段。某些数据库对象只存在于 SQL migration，
例如 partial/CONCURRENTLY/hot-path 索引；`backend/ent/schema/account.go:Indexes` 和
`user_subscription.go:Indexes` 的注释明确承认线上索引由 migration 创建。因此“两个模型保持
完全一一生成”不是本项目现实，SQL migration 最终决定线上 DDL。

### 生成文件边界

- `backend/ent/generate.go` 是手写入口，当前命令启用
  `sql/upsert,intercept,sql/execquery,sql/lock` 和 `int64` ID。
- `backend/ent/schema/**` 与 `backend/ent/generate.go` 可编辑；带
  `Code generated by ent, DO NOT EDIT.` 的其他 `backend/ent/**` 文件不可手改。
- 原生 SQL 与 Ent transaction 混用依赖 `sql/execquery` 生成能力；不要通过修改生成的
  `ent/tx.go`、`ent/tx_context.go` 等文件补行为。

## Repository 查询模式

### 何时用 Ent

普通实体读取和单表写入优先沿用 Ent：

```go
entity, err := client.User.Query().
    Where(dbuser.IDEQ(userID)).
    Only(ctx)
```

参考 `backend/internal/repository/user_repo.go:userRepository.GetByID`。使用 generated field
常量/predicate，不在 service 或 handler 拼接 SQL。关联预加载、upsert、`FOR UPDATE` 也使用
已启用的 Ent feature；例如 schema edge 及生成 query 的 lock API。

### 何时用原生 SQL

以下已有场景使用 `database/sql`，新增代码可在同类需求中沿用：

- 多字段条件扣减、幂等 claim、批量更新或需要 `RETURNING` 精确判断影响结果：
  `backend/internal/repository/usage_billing_repo.go:usageBillingRepository.Apply`。
- 大型聚合、窗口统计和数据库特定执行计划：`ops_repo_*.go`、
  `usage_log_repo_stats.go`。
- 需要把实体变更与 scheduler outbox 合成一条 SQL 的热路径：
  `backend/internal/repository/account_repo.go` 及其
  `account_repo_integration_test.go`。

原生 SQL 必须使用 `QueryContext`/`ExecContext`/`BeginTx` 并使用 PostgreSQL `$1` 参数，不能
拼接用户值。标识符只有在固定白名单/常量下才可格式化。所有 `Rows` 都应关闭并检查
`rows.Err()`；事务采用“defer rollback，成功后 commit 并清空/结束引用”的现有形状，参考
`usageBillingRepository.Apply`。

不要为了“统一 ORM”重写已经依赖 PostgreSQL 原子语义的 SQL；也不要为普通 CRUD 新建裸 SQL
而跳过 Ent 的软删除 interceptor、edge 和错误处理。

## 事务与 context 协议

### 外层事务拥有提交权

跨多个 repository 的原子操作由编排方开启 Ent transaction，并把 transaction 放进
context：

```go
transaction, err := entClient.Tx(ctx)
if err != nil {
    return err
}
defer func() { _ = transaction.Rollback() }()

transactionContext := ent.NewTxContext(ctx, transaction)
if err := firstRepository.Update(transactionContext, input); err != nil {
    return err
}
if err := secondRepository.Update(transactionContext, otherInput); err != nil {
    return err
}
return transaction.Commit()
```

真实编排见
`backend/internal/service/subscription_service.go:withSubscriptionUpdateTx` 和
`backend/internal/service/admin_group.go:AdminUpdateAPIKeyGroup` 的专属分组分支。已有 transaction
时应复用而不是嵌套；`withSubscriptionUpdateTx` 首先检查 `ent.TxFromContext(ctx)`。

### Repository 必须识别 transaction context

- Ent 操作通过 `backend/internal/repository/error_translate.go:clientFromContext` 取得
  `ent.TxFromContext(ctx).Client()`；否则使用 repository 默认 client。
- 自己可能开 transaction 的方法要先识别外层 transaction，并让外层负责 commit/rollback。
  `userRepository.Delete` 是明确例子；`userRepository.Create`/`Update` 还兼容
  `ent.ErrTxStarted`。
- `backend/internal/repository/user_repo_delete_atomicity_integration_test.go` 是此规则的回归证明：
  删除 API key 与 user 后回滚外层 transaction，两者必须一起恢复。独立开启并提前 commit
  会造成不可恢复的部分写入。

### Ent 与原生 SQL不是自动共享 transaction

`ProvideSQLDB` 暴露的是同一连接池，但在 `*sql.DB.BeginTx` 上开始的 transaction 不会因为共用
pool 就自动成为 Ent transaction，反之亦然。需要在 Ent transaction 内执行原生 SQL时，沿用
`backend/internal/repository/user_profile_identity_repo.go:txAwareSQLExecutor`，以及
`user_repo.go` 对该 helper 的调用；不要在 transaction context 中退回 base `*sql.DB` 写入。
纯原生 SQL 原子流程则可像 `usageBillingRepository.Apply` 一样完全由一个 `*sql.Tx` 完成。

### Commit 之后再做外部副作用

缓存失效、异步任务等不可回滚副作用应在成功 commit 后执行。真实说明见
`SubscriptionService.maybeInvalidateAssignmentCaches`：payment fulfillment 持有外层 transaction
时会延迟缓存失效，避免把未提交旧值重新加载进 cache。不要在 transaction 尚未提交时发布
“已成功”事件或刷新共享缓存，除非已有 outbox/补偿协议。

## 数据库错误边界

- Repository 负责把常见持久化错误翻译成业务可识别错误。
  `backend/internal/repository/error_translate.go:translatePersistenceError` 同时识别
  `ent.IsNotFound` 和 `sql.ErrNoRows`，并把 PostgreSQL `23505`/已知 duplicate 错误映射到传入的
  `*infraerrors.ApplicationError`，保留 `WithCause(err)`。
- 例如 `userRepository.GetByID` 映射 `service.ErrUserNotFound`，`Create`/`Update` 映射
  `service.ErrEmailExists`；`user_subscription_repo.go` 映射
  `ErrSubscriptionAlreadyExists`。不要让 handler 识别 `pq.Error`、constraint name 或
  `sql.ErrNoRows`。
- 未能稳定映射的基础设施错误仍可原样返回，service 再按业务语义包装或降级。现实中某些
  service 直接使用 Ent/SQL，因而会在本层检查 `ent.IsNotFound`、`ent.IsConstraintError` 或
  `sql.ErrNoRows`；例子见 `payment_fulfillment.go`、`auth_email_binding.go`、
  `ops_system_log_service.go` 和 `service/sql_errors.go`。这是真实技术耦合，不应扩散到普通
  handler。
- Handler 使用 `response.ErrorFrom` 将 application error 转 HTTP；数据库错误不应直接写入
  response。`backend/internal/handler/auth_handler.go:AuthHandler.Register` 是标准边界。

## 命名与数据约定

- 表和列使用 snake_case，实体表通常为复数：`users`、`api_keys`、`user_subscriptions`；关联
  表使用双方名称，如 `account_groups`、`user_allowed_groups`。Ent schema 用显式 table
  annotation 固定名称。
- 主键通常为 `BIGSERIAL`/`int64`；外键字段为 `<entity>_id`。关系的 `ON DELETE` 行为必须在
  migration 中明确，并由重要 schema 集成测试验证；参考
  `migrations_schema_integration_test.go:requireForeignKeyOnDelete`。
- 业务时刻字段通常使用 PostgreSQL `TIMESTAMPTZ`；`InitEnt` 把应用 timezone 注入 DSN，
  `TimeMixin` 及多数 Ent 时间字段显式声明该 schema type。新增同类字段时不要无依据切换成
  无时区 `timestamp`。
- 金额/费用列沿用邻近精度：余额常见 `DECIMAL(20,8)`，usage 成本常见
  `DECIMAL(20,10)`，倍率常见 `DECIMAL(10,4)`。不要在同一语义上随意换 precision。
- 手写索引多数使用 `idx_<table>_<columns>`，但历史和 Ent 生成命名存在例外（如
  `paymentorder_out_trade_no_unique`）。修改已有索引时引用真实名称，不根据约定猜测。
- 软删除实体的唯一性经常由 `WHERE deleted_at IS NULL` partial unique index 实现；参考
  `backend/migrations/016_soft_delete_partial_unique_indexes.sql` 和
  `schema/user_subscription.go` 注释。不要仅在 Ent field 上加 `Unique()` 就假定线上 partial
  约束已经更新。

## Migration 规则

### 文件、顺序与 forward-only

- Migration 位于 `backend/migrations/*.sql`，由
  `backend/migrations/migrations.go` 的 `//go:embed *.sql` 编入服务。
- Runner `backend/internal/repository/migrations_runner.go:applyMigrationsFS` 对完整文件名调用
  `sort.Strings`，执行顺序是**字典序**，不是解析出的整数序号。仓库真实存在字母后缀
  `108a_widen_auth_identity_migration_report_type.sql`，也存在相同数字前缀的多个文件（如
  `160_add_user_frozen_balance.sql` 与 `160_batch_image_provider_refs.sql`，以及多个 `174_*`）。
  新文件必须按完整文件名检查它位于所有依赖 migration 之后，不能假定前缀唯一。
- 当前 runner 没有 down migration 或 rollback-to-version 接口。已发布 schema 的修正通过新增
  migration 前向完成；不要重命名、删除或编辑已经可能应用的 SQL 文件。

### 启动执行、锁和记录

- `repository.InitEnt` 在创建 Ent client 前用 10 分钟 context 执行 migrations；失败会关闭
  driver 并阻止服务启动。
- `ApplyMigrations`/`applyMigrationsFS` 通过固定 ID 的 PostgreSQL
  `pg_try_advisory_lock` 轮询（`migrationsLockRetryInterval` 为 500ms），意图在多实例启动时
  串行化 migration；退出路径调用 `pg_advisory_unlock`。不要另写一套并行启动 migration。
- `schema_migrations(filename, checksum, applied_at)` 记录每个文件。Checksum 是 trim 后 SQL
  内容的 SHA-256；已记录文件 checksum 不一致时启动失败。
- `migrationChecksumCompatibilityRules` 只兼容少数历史上已误改文件的已知 checksum 组合。
  新 migration 不得加入宽泛豁免来绕过不可变性；正确修复方式是恢复原文件并新增 forward
  migration。
- `ensureAtlasBaselineAligned` 会在 legacy `schema_migrations` 存在时创建/填充空的
  `atlas_schema_revisions` baseline。这是兼容桥接逻辑，不表示 Atlas 取代了当前 embedded SQL
  runner。

### 默认 transaction migration

普通 `.sql` 文件在一个 `database/sql` transaction 内执行，DDL 成功和写入
`schema_migrations` 一起 commit；任一步失败都会 rollback。新 migration 优先沿用已有幂等
写法（`ADD COLUMN IF NOT EXISTS`、`CREATE TABLE/INDEX IF NOT EXISTS` 等），例如
`177_add_subscription_plan_currency.sql`。但并非每个历史文件都语法级幂等，不能据此声称可以
任意手工重复整段 SQL；以 runner 的记录和 transaction 为准。

### `*_notx.sql`

`CREATE/DROP INDEX CONCURRENTLY` 不能在 PostgreSQL transaction block 中执行，必须放进
以 `_notx.sql` 结尾的文件。`validateMigrationExecutionMode` 施加真实限制：

- 普通 migration 出现 `CONCURRENTLY` 会被拒绝。
- `_notx.sql` 只能包含 `CREATE INDEX CONCURRENTLY` 或 `DROP INDEX CONCURRENTLY`。
- CREATE 必须有 `IF NOT EXISTS`，DROP 必须有 `IF EXISTS`。
- 禁止 `BEGIN`、`COMMIT`、`ROLLBACK`，也禁止混入普通 DDL/DML。
- Runner 按分号拆分逐条直接执行，最后单独记录 checksum；因此非 transaction 文件尤其要保证
  可重试。参考 `174_add_usage_logs_api_key_latest_ip_index_notx.sql` 和
  `175a_add_ops_system_logs_host_index_notx.sql`。

部分历史 CONCURRENTLY migration 有失败重试前清理无效索引或数据 precheck；具体白名单在
`prepareNonTransactionalMigration`。新增特殊恢复逻辑必须有针对性测试，不能把所有 index
静默 drop/recreate。

## 验证与代表性测试

- `backend/internal/repository/migrations_runner_checksum_test.go`：checksum 兼容白名单必须精确，
  未知组合拒绝。
- `backend/internal/repository/migrations_runner_notx_test.go`：`_notx` validator、逐语句执行、
  无效索引恢复和 precheck。
- `backend/internal/repository/migrations_runner_extra_test.go`：checksum mismatch、advisory lock、
  Atlas baseline 等错误分支。
- `backend/internal/repository/migrations_schema_integration_test.go`：重复执行 migration 的幂等性
  及真实 PostgreSQL 列、索引、constraint/foreign key 形状。
- `backend/internal/repository/integration_harness_test.go`：`//go:build integration` 下用
  testcontainers 启动 PostgreSQL/Redis 并先调用 `ApplyMigrations`。
- `backend/internal/repository/usage_billing_repo_unit_test.go` 使用 `go-sqlmock` 锁定原生 SQL 的
  transaction 和参数；`user_repo_delete_atomicity_integration_test.go` 锁定跨 repository
  transaction context。

可靠命令来自 `backend/Makefile`：

```bash
make -C backend generate
make -C backend test-unit
make -C backend test-integration
make -C backend test
```

`test-integration` 需要 Docker；普通 `test` 还会运行 `golangci-lint run ./...`。针对文档或小型
数据库改动可先跑对应 package/test，但 schema/migration 变更完成前至少应覆盖相关 migration
runner 和 repository 测试。

## 反模式

- 不编辑已应用 migration 来“修正历史”，也不通过扩大 checksum 白名单掩盖修改。
- 不假定 migration 前缀是唯一整数或按创建时间执行；runner 只看完整文件名字典序。
- 不把 `CONCURRENTLY` 放进普通 migration，也不在 `_notx.sql` 混入表变更或 transaction 控制。
- 不在外层 transaction context 中使用 base Ent client/base `*sql.DB` 写入并提前 commit。
- 不在 handler 解析 PostgreSQL/Ent 错误；repository/service 先转换为业务错误。
- 不绕过 `SoftDeleteMixin` 做普通读取，也不忘记原生 SQL 自己补 `deleted_at IS NULL`。
- 不手改 Ent 生成文件，不用 Ent auto-migration 替代 embedded SQL migration。
