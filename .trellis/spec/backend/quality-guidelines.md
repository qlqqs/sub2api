# 后端质量规范

后端以 `backend/go.mod`、`backend/.golangci.yml`、`backend/Makefile` 和根 `Makefile` 为可执行质量契约。保持 Handler -> Service interface -> Repository 的依赖方向，先运行最窄测试，再运行对应质量门。

## 静态检查与依赖边界

golangci-lint v2 默认不启用 linter，项目显式启用 `depguard`、`errcheck`、`gosec`、`govet`、`ineffassign`、`staticcheck`、`unused`，formatter 使用 `gofmt`。gofmt rewrite 将 `interface{}` 改为 `any`、`a[b:len(a)]` 改为 `a[b:]`。

- Service 不得导入 `internal/repository`、GORM 或 go-redis。现有例外仅限 `ops_aggregation_service.go`、`ops_alert_evaluator_service.go`、`ops_cleanup_service.go`、`ops_metrics_collector.go`、`ops_scheduled_report_service.go` 与 `internal/service/wire.go`；不得把例外复制到新文件。
- `internal/handler/**` 不得直接导入 Repository、GORM 或 go-redis，持久化能力通过 Service 接口和 Wire 注入。
- `errcheck` 检查 type assertion，并禁用默认排除列表；除配置明确列出的打印/复制函数外，错误必须处理。`ineffassign` 同时检查逃逸 error。
- `gosec` 仅报告 high severity/high confidence；项目排除 G101、G103、G104、G109、G115、G201、G202、G301、G302、G304、G306、G404。排除不代表允许凭证泄露或不安全路径。
- `staticcheck` 开启全部检查，仅关闭历史注释/命名兼容项 ST1000、ST1003、ST1020、ST1021、ST1022。
- `unused.generated-is-used: true` 为 Ent 生成代码保留，不能用于掩盖手写死代码。

## 禁止模式

- Handler 直接访问数据库/Redis，或 Service 新增对具体 Repository 包的导入。
- 手工编辑 `backend/ent/**` 中标有 `Code generated by ent, DO NOT EDIT` 的文件，或手工编辑 `backend/cmd/server/wire_gen.go`。
- 修改已有迁移文件而没有处理 checksum 兼容；默认新增顺序迁移。历史兼容白名单由 `internal/repository/migrations_runner_checksum_test.go` 约束。
- 用无 build tag 的测试依赖真实 Docker/PostgreSQL/Redis，导致默认测试环境不稳定。
- 在测试中用固定 sleep 等待并发状态；优先条件轮询、channel、deadline 和 eventually 断言。
- 为通过检查而扩大 lint ignore、depguard 例外或安全排除。例外必须最小化并说明现有架构原因。
- 在无回归价值时大量复制实现细节测试；优先覆盖边界契约、错误分支、事务原子性、协议格式和迁移结果。

## 生成代码与迁移验证

生成入口由 `backend/Makefile:generate` 固定：

```bash
cd backend
make generate
```

该目标依次执行 `go generate ./ent` 和 `go generate ./cmd/server`。`backend/ent/generate.go` 使用 Ent 的 `sql/upsert,intercept,sql/execquery,sql/lock` feature 与 `int64` ID；`backend/cmd/server/wire.go` 带 `wireinject` build tag，生成生产依赖图。修改 Ent schema、ProviderSet、构造函数或 Wire binding 后必须重新生成，并检查 diff 只包含预期产物。

迁移测试分三层：

- `backend/migrations/*_test.go` 直接读取嵌入 SQL，快速验证关键 SQL 片段和保护条件。
- `backend/internal/repository/migrations_runner_*_test.go` 验证 runner 顺序、checksum、非事务迁移等纯逻辑。
- `backend/internal/repository/migrations_schema_integration_test.go` 在真实 PostgreSQL 上重复 `ApplyMigrations`，验证幂等、表、列、索引、约束、默认值和外键动作。

新增或修改迁移必须至少覆盖对应层，涉及数据库实际语义时必须加 integration 断言。

## 测试层级与命令覆盖范围

测试 build tag 放在文件第一行，项目使用 `unit`、`integration`、`e2e`：

| 命令 | 实际范围 |
|---|---|
| `cd backend && go test ./...` | 运行所有无 build tag 的后端测试；不包含仅标记 unit/integration/e2e 的文件。 |
| `cd backend && make test` | 先执行上述默认测试，再执行 `golangci-lint run ./...`；这是根 `make test-backend` 使用的后端门。 |
| `cd backend && make test-unit` | `go test -tags=unit ./...`，运行 unit-tagged 与无 tag 测试，不启动 Testcontainers。 |
| `cd backend && make test-integration` | `go test -tags=integration ./...`，运行 integration-tagged 与无 tag 测试；需要 Docker。 |
| `cd backend && make test-e2e` | 委托 `./scripts/e2e-test.sh`；当前工作树未找到该脚本，补齐前此 target 无法执行。 |
| `cd backend && make test-e2e-local` | `go test -tags=e2e -v -timeout=300s ./internal/integration/...`，只跑本地集成目录的 E2E 包。 |
| 根目录 `make test` | `make -C backend test`，随后前端 lint、typecheck 和关键 Vitest；不是所有 build tag 的总和。 |

`backend/internal/repository/integration_harness_test.go:TestMain` 使用 Testcontainers 启动 `postgres:18.1-alpine3.23` 和 `redis:8.4-alpine`，应用真实迁移后共享连接。无 Docker 时本地整体跳过，CI 中则失败；因此不能把“本地无 Docker 返回成功”误判为 integration 已覆盖。

数据库测试优先使用 `testEntTx`/`IntegrationDBSuite`，每例自动回滚；Redis 使用 `testRedis` 的命名空间 hook 与 cleanup。`testEntSQLTx` 已弃用并主动失败，因为基于 `*sql.Tx` 的 Ent client 再调用 `client.Tx()` 会 panic。需要测试内部自建事务的代码时使用 `testEntClient`，并显式清理真实写入。

提交前选择最窄且足够的测试；错误协议、并发、账务、迁移和持久化语义的修改不能只依赖编译或默认无 tag 测试。

## 代码评审清单

- 依赖方向是否满足 depguard；新增例外是否可通过接口注入消除。
- 所有错误是否被处理、包装或有意忽略；客户端错误与日志是否遵守错误/脱敏规范。
- 是否存在手改生成文件；schema/Wire 变化后是否执行 `make generate` 并检查生成 diff。
- 测试是否选择了正确 build tag；默认、unit、integration、e2e 的覆盖范围是否被准确说明。
- 集成测试是否事务/命名空间隔离，Docker 缺失时的跳过是否被误当作通过。
- 迁移是否保持顺序、checksum 和幂等，且对实际 schema 结果有断言。
- 至少运行受影响包测试；跨层改动运行 `cd backend && make test`，数据库语义改动再运行 `make test-integration`，完整交付按根 `make test` 验证前后端门。
